# Application Monitoring Techniques and Tools

## Overview

Modern applications must perform reliably under pressure. Monitoring application health and performance ensures your systems remain reliable and responsive, even under real-world production conditions. This guide explores key techniques and tools for monitoring applications, with a focus on Azure DevOps and GitHub Actions integration.

## Core Monitoring Techniques

### 1. Real-Time Monitoring

Real-time monitoring involves continuously tracking the performance, health, and status of various application server-side components.

**Key Features:**
- Live metrics assessment
- Critical threshold alerts (CPU usage, memory consumption)
- Interactive dashboards
- Instant anomaly detection

**Azure Monitor Implementation:**

Azure Monitor is a comprehensive solution for monitoring applications running in Microsoft Azure.

```yaml
# Azure DevOps Pipeline - Enable Azure Monitor
trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

steps:
- task: AzureCLI@2
  inputs:
    azureSubscription: 'your-subscription'
    scriptType: 'bash'
    scriptLocation: 'inlineScript'
    inlineScript: |
      # Enable Application Insights
      az monitor app-insights component create \
        --app myapp-insights \
        --location eastus \
        --resource-group myResourceGroup \
        --application-type web
```

**GitHub Actions Example:**

```yaml
name: Deploy with Monitoring

on:
  push:
    branches: [ main ]

jobs:
  deploy-and-monitor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Azure Login
        uses: azure/login@v2
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}
      
      - name: Configure Azure Monitor
        run: |
          az monitor metrics alert create \
            --name high-cpu-alert \
            --resource-group myResourceGroup \
            --scopes /subscriptions/{subscription-id}/resourceGroups/myResourceGroup \
            --condition "avg Percentage CPU > 80" \
            --description "Alert when CPU exceeds 80%"
```

**Example Scenario:**
An alert from Azure Monitor notifies you of an extreme increase in CPU usage. Addressing it promptly prevents user delays and keeps the system operating smoothly.

### 2. Log Monitoring

Log monitoring is the process of collecting, analyzing, and tracking log data from applications, servers, databases, and network devices.

**Purpose:**
- Detect errors and exceptions
- Troubleshoot issues
- Gain insights into system performance
- Identify security threats
- Uncover patterns and root causes

**Azure Log Analytics:**

Log Analytics, a tool in the Azure Monitor portal, allows you to edit and run log queries to analyze collected logs effectively.

```yaml
# Azure DevOps - Configure Log Analytics Workspace
steps:
- task: AzureCLI@2
  displayName: 'Setup Log Analytics'
  inputs:
    azureSubscription: 'your-subscription'
    scriptType: 'bash'
    scriptLocation: 'inlineScript'
    inlineScript: |
      # Create Log Analytics Workspace
      az monitor log-analytics workspace create \
        --resource-group myResourceGroup \
        --workspace-name myLogWorkspace \
        --location eastus
      
      # Link to Application Insights
      az monitor app-insights component connect-workspace \
        --app myapp-insights \
        --resource-group myResourceGroup \
        --workspace myLogWorkspace
```

**GitHub Actions - Log Collection:**

```yaml
name: Collect and Analyze Logs

on:
  schedule:
    - cron: '0 */6 * * *'  # Every 6 hours

jobs:
  log-analysis:
    runs-on: ubuntu-latest
    steps:
      - name: Query Azure Logs
        uses: azure/CLI@v2
        with:
          inlineScript: |
            # Query logs for errors in the last 24 hours
            az monitor log-analytics query \
              --workspace myLogWorkspace \
              --analytics-query "
                requests
                | where timestamp > ago(24h)
                | where success == false
                | summarize count() by resultCode, bin(timestamp, 1h)
                | order by timestamp desc
              " \
              --output table
      
      - name: Upload Log Report
        uses: actions/upload-artifact@v4
        with:
          name: log-analysis
          path: logs/
```

**Example Use Case:**
A company uses Azure Monitor Log Analytics to track API request logs, identify slow response times, and automatically trigger alerts for failures. By centralizing logs, you can easily spot recurring issues such as timeout errors during peak hours.

### 3. Application Performance Monitoring (APM)

APM is a set of practices for monitoring and managing the performance and availability of software applications.

**Key Metrics Tracked:**
- Response times
- Error rates
- Transaction traces
- Resource usage (CPU, memory, disk I/O)
- Database query performance
- API endpoint latency

**Azure Application Insights:**

Application Insights plays a central role in APM by enabling real-time collection, visualization, and analysis of performance data.

```yaml
# Azure DevOps - Deploy with Application Insights
trigger:
  - main

variables:
  azureSubscription: 'your-subscription'
  appName: 'mywebapp'

stages:
- stage: Deploy
  jobs:
  - job: DeployWithMonitoring
    steps:
    - task: AzureWebApp@1
      inputs:
        azureSubscription: $(azureSubscription)
        appName: $(appName)
        package: '$(System.DefaultWorkingDirectory)/**/*.zip'
    
    - task: AzureCLI@2
      displayName: 'Configure Application Insights'
      inputs:
        azureSubscription: $(azureSubscription)
        scriptType: 'bash'
        scriptLocation: 'inlineScript'
        inlineScript: |
          # Get instrumentation key
          INSTRUMENTATION_KEY=$(az monitor app-insights component show \
            --app myapp-insights \
            --resource-group myResourceGroup \
            --query instrumentationKey -o tsv)
          
          # Update app settings
          az webapp config appsettings set \
            --name $(appName) \
            --resource-group myResourceGroup \
            --settings APPINSIGHTS_INSTRUMENTATIONKEY=$INSTRUMENTATION_KEY
```

**GitHub Actions - APM Integration:**

```yaml
name: Deploy with APM

on:
  push:
    branches: [ main ]

env:
  AZURE_WEBAPP_NAME: mywebapp
  RESOURCE_GROUP: myResourceGroup

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Build application
        run: npm run build
      
      - name: Azure Login
        uses: azure/login@v2
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}
      
      - name: Deploy to Azure Web App
        uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ env.AZURE_WEBAPP_NAME }}
          package: .
      
      - name: Configure Performance Alerts
        run: |
          # Create alert for slow response times
          az monitor metrics alert create \
            --name slow-response-alert \
            --resource-group ${{ env.RESOURCE_GROUP }} \
            --scopes /subscriptions/${{ secrets.SUBSCRIPTION_ID }}/resourceGroups/${{ env.RESOURCE_GROUP }}/providers/Microsoft.Web/sites/${{ env.AZURE_WEBAPP_NAME }} \
            --condition "avg requests/duration > 3000" \
            --description "Alert when average response time exceeds 3 seconds" \
            --evaluation-frequency 5m \
            --window-size 15m
          
          # Create alert for error rate
          az monitor metrics alert create \
            --name high-error-rate-alert \
            --resource-group ${{ env.RESOURCE_GROUP }} \
            --scopes /subscriptions/${{ secrets.SUBSCRIPTION_ID }}/resourceGroups/${{ env.RESOURCE_GROUP }}/providers/Microsoft.Web/sites/${{ env.AZURE_WEBAPP_NAME }} \
            --condition "avg requests/failed > 5" \
            --description "Alert when error rate exceeds threshold"
```

**Example Scenario:**
A development team uses Azure Monitor Application Insights to track the performance of their web application, identifying slow API responses and database queries. The team sets up real-time alerts in Application Insights to automatically notify them when an error rate exceeds a certain threshold, allowing for immediate troubleshooting.

## Advanced Monitoring Patterns

### Continuous Monitoring in CI/CD

**Azure DevOps Multi-Stage Pipeline:**

```yaml
trigger:
  - main

stages:
- stage: Build
  jobs:
  - job: BuildJob
    steps:
    - script: npm install && npm run build
      displayName: 'Build Application'

- stage: Test
  jobs:
  - job: TestJob
    steps:
    - script: npm test
      displayName: 'Run Tests'

- stage: Deploy
  jobs:
  - deployment: DeployWeb
    environment: 'production'
    strategy:
      runOnce:
        deploy:
          steps:
          - task: AzureWebApp@1
            inputs:
              azureSubscription: 'your-subscription'
              appName: 'mywebapp'

- stage: Monitor
  dependsOn: Deploy
  jobs:
  - job: PostDeploymentMonitoring
    steps:
    - task: AzureCLI@2
      displayName: 'Check Application Health'
      inputs:
        azureSubscription: 'your-subscription'
        scriptType: 'bash'
        scriptLocation: 'inlineScript'
        inlineScript: |
          # Wait for deployment to stabilize
          sleep 60
          
          # Query Application Insights for errors
          az monitor app-insights query \
            --app myapp-insights \
            --analytics-query "
              requests
              | where timestamp > ago(5m)
              | summarize 
                  TotalRequests = count(),
                  FailedRequests = countif(success == false),
                  AvgDuration = avg(duration)
              | extend ErrorRate = (FailedRequests * 100.0) / TotalRequests
            "
          
          # Fail pipeline if error rate is too high
          ERROR_RATE=$(az monitor app-insights query ... | jq '.tables[0].rows[0][3]')
          if (( $(echo "$ERROR_RATE > 5" | bc -l) )); then
            echo "Error rate too high: $ERROR_RATE%"
            exit 1
          fi
```

**GitHub Actions - Progressive Deployment with Monitoring:**

```yaml
name: Progressive Deployment

on:
  push:
    branches: [ main ]

jobs:
  deploy-canary:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy to Canary Slot
        uses: azure/webapps-deploy@v3
        with:
          app-name: mywebapp
          slot-name: canary
          package: .
      
      - name: Monitor Canary Performance
        run: |
          echo "Monitoring canary deployment for 10 minutes..."
          sleep 600
          
          # Query metrics
          CANARY_ERRORS=$(az monitor metrics list \
            --resource /subscriptions/.../slots/canary \
            --metric "Http5xx" \
            --start-time $(date -u -d '10 minutes ago' +%Y-%m-%dT%H:%M:%SZ) \
            --interval PT1M \
            --aggregation Total \
            --query 'value[0].timeseries[0].data[-1].total' -o tsv)
          
          if [ "$CANARY_ERRORS" -gt 10 ]; then
            echo "Canary showing errors, aborting deployment"
            exit 1
          fi
      
      - name: Swap to Production
        if: success()
        run: |
          az webapp deployment slot swap \
            --name mywebapp \
            --resource-group myResourceGroup \
            --slot canary \
            --target-slot production

  post-deployment-validation:
    needs: deploy-canary
    runs-on: ubuntu-latest
    steps:
      - name: Validate Production Health
        run: |
          # Run synthetic tests
          curl -f https://mywebapp.azurewebsites.net/health || exit 1
          
          # Check Application Insights
          az monitor app-insights query \
            --app myapp-insights \
            --analytics-query "
              availabilityResults
              | where timestamp > ago(5m)
              | summarize SuccessRate = avg(success) * 100
            "
```

## Monitoring Dashboards and Visualization

### Custom Azure Dashboard via Pipeline

```yaml
- task: AzureCLI@2
  displayName: 'Create Monitoring Dashboard'
  inputs:
    azureSubscription: 'your-subscription'
    scriptType: 'bash'
    scriptLocation: 'inlineScript'
    inlineScript: |
      # Create dashboard JSON
      cat > dashboard.json <<EOF
      {
        "properties": {
          "lenses": [
            {
              "parts": [
                {
                  "metadata": {
                    "type": "Extension/Microsoft_Azure_Monitoring/PartType/MetricsChartPart",
                    "settings": {
                      "content": {
                        "metrics": [
                          {
                            "resourceMetadata": {
                              "id": "/subscriptions/.../providers/Microsoft.Web/sites/mywebapp"
                            },
                            "name": "Requests",
                            "aggregationType": "Total"
                          }
                        ]
                      }
                    }
                  }
                }
              ]
            }
          ]
        }
      }
      EOF
      
      # Deploy dashboard
      az portal dashboard create \
        --name "Production Monitoring" \
        --resource-group myResourceGroup \
        --input-path dashboard.json
```

## Best Practices

### 1. Establish Baseline Metrics
- Collect performance data during normal operations
- Set realistic alert thresholds based on historical data
- Regularly review and adjust baselines

### 2. Implement Comprehensive Alerting
- Configure alerts for critical metrics (CPU, memory, response time)
- Use action groups for notification routing
- Avoid alert fatigue with smart grouping

### 3. Centralize Logging
- Use structured logging formats (JSON)
- Implement correlation IDs for distributed tracing
- Retain logs based on compliance requirements

### 4. Automate Monitoring Configuration
- Define monitoring as code in your pipelines
- Version control monitoring configurations
- Test monitoring setup in non-production environments

### 5. Regular Review and Optimization
- Conduct monthly performance reviews
- Optimize slow queries and endpoints
- Remove obsolete metrics and alerts

## Conclusion

Understanding and implementing monitoring techniques and tools equips you to detect and resolve performance issues before they affect users. By integrating Azure Monitor, Application Insights, and Log Analytics into your Azure DevOps and GitHub Actions workflows, you'll ensure your applications remain reliable, high-performing, and ready to meet the demands of dynamic production environments.

### Key Takeaways

- **Real-time monitoring** provides immediate visibility into system health
- **Log monitoring** helps identify patterns and troubleshoot historical issues
- **Application Performance Monitoring** offers deep insights into application behavior
- **CI/CD integration** enables continuous monitoring throughout the deployment lifecycle
- **Automation** ensures consistent monitoring practices across all environments
