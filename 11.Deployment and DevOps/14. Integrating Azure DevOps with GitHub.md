# Integrating Azure DevOps with GitHub Repositories

## Overview

Azure DevOps provides a powerful CI/CD environment that offers enhanced visibility and control compared to GitHub's native workflows. This guide demonstrates how to connect a GitHub repository to Azure DevOps for automated deployment workflows.

## Why Use Azure DevOps?

While GitHub provides continuous integration and deployment capabilities, Azure DevOps offers:

- Enhanced visibility into CI/CD pipelines
- More configuration options
- Advanced monitoring and reporting features
- Powerful workflow management tools

## Prerequisites

- A GitHub repository with your application code
- An Azure DevOps account (accessible at `dev.azure.com`)
- An Azure subscription for hosting your application

## Setup Process

### 1. Create an Azure DevOps Organization

Organizations in Azure DevOps serve as containers for your CI/CD pipelines and projects.

**Example:**
- Organization name: `devproject2025`
- Multiple pipelines can exist within a single organization

### 2. Connect GitHub Repository to Azure DevOps

Navigate to your Azure DevOps project settings to establish the connection:

1. Go to **Settings** (bottom of the sidebar)
2. Under **Boards**, select **GitHub connections**
3. Connect your GitHub account
4. Select the specific repository to integrate

**Example:**
- Repository name: `Azure Web App`
- Once connected, any GitHub events will trigger the Azure DevOps pipeline

### 3. Configure Build Agent

Azure DevOps requires an agent to execute pipeline tasks. You have two options:

**Option A: Azure-Hosted Agent**
- Managed by Azure
- No local setup required

**Option B: Self-Hosted Agent**
- Runs on your local machine
- Provides more control over the build environment

**Example Configuration:**
```yaml
pool:
  name: default  # Name of your agent pool
```

## Creating the Azure Pipeline

### Navigate to Pipelines

1. From the sidebar, select **Pipelines**
2. Create a new pipeline or edit an existing one
3. Define the pipeline using YAML format

### Example YAML Pipeline Configuration

Here's a complete pipeline configuration for a Blazor WebAssembly application:

```yaml
# Trigger: Define when the pipeline should run
trigger:
  branches:
    include:
      - main  # Run on commits to the main branch
  paths:
    include:
      - Azure Web App/*  # Monitor specific paths

# Agent Pool: Specify where the build runs
pool:
  name: default  # Your configured agent pool

# Variables: Define build settings
variables:
  buildConfiguration: 'Release'

# Steps: Define the build and deployment process
steps:
  # Step 1: Restore NuGet packages
  - task: DotNetCoreCLI@2
    displayName: 'Restore dependencies'
    inputs:
      command: 'restore'
      projects: '**/*.csproj'

  # Step 2: Build the project
  - task: DotNetCoreCLI@2
    displayName: 'Build project'
    inputs:
      command: 'build'
      projects: '**/*.csproj'
      arguments: '--configuration $(buildConfiguration)'

  # Step 3: Publish the application
  - task: DotNetCoreCLI@2
    displayName: 'Publish application'
    inputs:
      command: 'publish'
      publishWebProjects: true
      arguments: '--configuration $(buildConfiguration) --output $(Build.ArtifactStagingDirectory)'
      zipAfterPublish: true

  # Step 4: Deploy to Azure Web App
  - task: AzureWebApp@1
    displayName: 'Deploy to Azure Web App'
    inputs:
      azureSubscription: 'azureconnect-msft-webapp'
      appType: 'webApp'
      appName: 'Microsoft-Dev-Web-App'
      package: '$(Build.ArtifactStagingDirectory)/**/*.zip'
```

## Pipeline Workflow Explanation

### Task Breakdown

**1. Restore Dependencies**
```yaml
- task: DotNetCoreCLI@2
  inputs:
    command: 'restore'
    projects: '**/*.csproj'
```
- Runs `dotnet restore` command
- Ensures all NuGet packages are available

**2. Build Application**
```yaml
- task: DotNetCoreCLI@2
  inputs:
    command: 'build'
    projects: '**/*.csproj'
    arguments: '--configuration Release'
```
- Compiles the application
- Uses Release configuration for optimization

**3. Publish Application**
```yaml
- task: DotNetCoreCLI@2
  inputs:
    command: 'publish'
    publishWebProjects: true
    arguments: '--output $(Build.ArtifactStagingDirectory)'
```
- Creates deployment-ready package
- Outputs to staging directory
- Automatically zips the application

**4. Deploy to Azure**
```yaml
- task: AzureWebApp@1
  inputs:
    azureSubscription: 'azureconnect-msft-webapp'
    appType: 'webApp'
    appName: 'Microsoft-Dev-Web-App'
    package: '$(Build.ArtifactStagingDirectory)/**/*.zip'
```
- Uploads zip package to Azure Web App
- Azure automatically extracts and deploys the application

## Running the Pipeline

### Automatic Triggers

The pipeline runs automatically when:
- Code is pushed to the monitored branch (e.g., `main`)
- Pull requests are merged
- Any configured trigger event occurs

### Manual Execution

You can also run the pipeline manually:

1. Navigate to **Pipelines**
2. Select your pipeline (e.g., "Azure Pipelines")
3. Click **Run pipeline**
4. The job will be queued and executed

**Example Output:**
```

Running for 11 seconds...
✓ Restore dependencies - Completed
✓ Build project - Completed
```

## Monitoring and Visibility

Azure DevOps provides comprehensive monitoring capabilities:

- **Real-time status**: Watch pipeline execution in real-time
- **Detailed logs**: View logs for each step
- **Success indicators**: Green checkmarks for successful tasks
- **Execution time**: Track how long each step takes
- **Historical data**: Review past pipeline runs

## Benefits of This Integration

1. **Automated Deployments**: Push to GitHub, automatic deployment to Azure
2. **Manual Control**: Run pipelines on-demand when needed
3. **Visibility**: Monitor every step of the build and deployment process
4. **Flexibility**: Configure complex workflows with YAML
5. **Reliability**: Ensure consistent build and deployment processes

## Complete Workflow Example

Here's the end-to-end process:

1. **Developer pushes code** to GitHub repository
2. **GitHub webhook** notifies Azure DevOps
3. **Pipeline triggers** automatically
4. **Agent picks up** the job from the queue
5. **Code is pulled** from GitHub
6. **Dependencies restored** via `dotnet restore`
7. **Application built** with Release configuration
8. **Project published** and zipped
9. **Package deployed** to Azure Web App
10. **Application goes live** for users

## Summary

Azure DevOps integration with GitHub provides a robust CI/CD solution that combines the best of both platforms. You maintain your source code in GitHub while leveraging Azure DevOps' powerful pipeline and monitoring capabilities.

**Key Steps:**
1. Create Azure DevOps organization and project
2. Establish service connection to GitHub repository
3. Configure Azure pipeline using YAML
4. Monitor and manage deployments from Azure DevOps interface

This approach is ideal when you need advanced workflow management, detailed monitoring, and enterprise-grade deployment pipelines.
