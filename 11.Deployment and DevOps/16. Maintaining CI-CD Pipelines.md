# Maintaining CI/CD Pipelines: Best Practices Guide

## Overview

Once you've created a CI/CD pipeline, it's important to periodically review and maintain it to ensure reliability and take advantage of new features. This guide covers essential maintenance techniques for Azure DevOps pipelines.

---

## Initial Setup Example

### Project Structure
- **Application**: Simple Blazor WebAssembly application (using default template)
- **Repository**: GitHub repository named "Azure Web App"
- **CI/CD Platform**: Azure DevOps
- **Deployment Target**: Azure Web App Service

### Repository Configuration
1. Create local Git repository
2. Add and commit application code
3. Create GitHub repository
4. Push local code to GitHub
5. Connect GitHub to Azure DevOps project

## Connecting GitHub to Azure DevOps

### Navigation Path (Current Azure DevOps UI)
```
Organization Settings → GitHub connections → Add connection
```

### Steps:
1. Navigate to your **Organization Settings** (bottom-left gear icon)
2. Under **General**, select **GitHub connections**
3. Click **+ Add connection**
4. Authorize Azure Pipelines to access your GitHub repository
5. Select the specific repository (e.g., "Azure Web App")
6. Complete the connection setup

## Maintaining Your Pipeline YAML

### Accessing Your Pipeline
1. Go to **Pipelines** in the left navigation
2. Select your pipeline from the list
3. Click the **three dots (⋮)** menu
4. Select **Edit**

This opens the YAML editor where you can review and modify your pipeline configuration.

### Example 1: Updating CLI Versions

#### Before (Outdated):
```yaml
steps:
- task: DotNetCoreCLI@1
  inputs:
    command: 'restore'
    projects: '**/*.csproj'

- task: DotNetCoreCLI@1
  inputs:
    command: 'build'
    projects: '**/*.csproj'
```

#### After (Updated):
```yaml
steps:
- task: DotNetCoreCLI@2
  inputs:
    command: 'restore'
    projects: '**/*.csproj'

- task: DotNetCoreCLI@2
  inputs:
    command: 'build'
    projects: '**/*.csproj'
```

**Why Update?**
- Access to bug fixes
- New features and improvements
- Better performance and security

### Validation Process
1. Make changes in the YAML editor
2. Click **Validate** (top-right toolbar)
3. Azure DevOps checks for syntax errors
4. Click **Save** to commit changes
5. Add a commit message describing your updates

## Adding Automated Rollback Capabilities

### Basic Pipeline Structure (Before)

```yaml
trigger:
- main

pool:
  vmImage: 'ubuntu-latest'

variables:
  buildConfiguration: 'Release'

steps:
- task: DotNetCoreCLI@2
  displayName: 'Restore packages'
  inputs:
    command: 'restore'
    projects: '**/*.csproj'

- task: DotNetCoreCLI@2
  displayName: 'Build project'
  inputs:
    command: 'build'
    projects: '**/*.csproj'
    arguments: '--configuration $(buildConfiguration)'

- task: AzureWebApp@1
  displayName: 'Deploy to Azure Web App'
  inputs:
    azureSubscription: 'Your-Subscription'
    appType: 'webApp'
    appName: 'microsoft-dev-webapp'
    package: '$(System.DefaultWorkingDirectory)/**/*.zip'
```

### Enhanced Pipeline with Stages and Rollback (After)

```yaml
trigger:
- main

pool:
  vmImage: 'ubuntu-latest'

variables:
  buildConfiguration: 'Release'

stages:
- stage: Build
  displayName: 'Build Stage'
  jobs:
  - job: BuildJob
    displayName: 'Build Application'
    steps:
    - task: DotNetCoreCLI@2
      displayName: 'Restore dependencies'
      inputs:
        command: 'restore'
        projects: '**/*.csproj'
    
    - task: DotNetCoreCLI@2
      displayName: 'Build project'
      inputs:
        command: 'build'
        projects: '**/*.csproj'
        arguments: '--configuration $(buildConfiguration)'
    
    - task: DotNetCoreCLI@2
      displayName: 'Publish application'
      inputs:
        command: 'publish'
        publishWebProjects: true
        arguments: '--configuration $(buildConfiguration) --output $(Build.ArtifactStagingDirectory)'
        zipAfterPublish: true
    
    - task: PublishBuildArtifacts@1
      displayName: 'Publish artifacts'
      inputs:
        PathtoPublish: '$(Build.ArtifactStagingDirectory)'
        ArtifactName: 'drop'

- stage: Deploy
  displayName: 'Deploy to Production'
  dependsOn: Build
  condition: succeeded()
  jobs:
  - deployment: DeployWeb
    displayName: 'Deploy to Azure Web App'
    environment: 'production'
    strategy:
      runOnce:
        deploy:
          steps:
          - task: DownloadBuildArtifacts@1
            displayName: 'Download artifacts'
            inputs:
              buildType: 'current'
              downloadType: 'single'
              artifactName: 'drop'
              downloadPath: '$(System.ArtifactsDirectory)'
          
          - task: AzureWebApp@1
            displayName: 'Deploy to Azure'
            inputs:
              azureSubscription: 'Your-Azure-Subscription'
              appType: 'webApp'
              appName: 'microsoft-dev-webapp'
              package: '$(System.ArtifactsDirectory)/drop/**/*.zip'
        
        on:
          failure:
            steps:
            - script: |
                echo "Deployment failed! Rolling back to previous build..."
                # Add rollback logic here:
                # 1. Retrieve last successful build artifact
                # 2. Deploy previous version
                # 3. Send notification to team
              displayName: 'Rollback on Failure'
```

### Key Improvements:
1. **Separation of Concerns**: Build and Deploy are separate stages
2. **Artifact Management**: Build artifacts are published and downloaded
3. **Failure Handling**: Automatic detection and rollback capability
4. **Environment Tracking**: Deployment to named environments (production)

## Automated Dependency Management with Dependabot

### Setting Up Dependabot

#### File Location:
```
project-root/
└── .github/
    └── dependabot.yml
```

#### Configuration Example:

```yaml
version: 2
updates:
  - package-ecosystem: "nuget"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    reviewers:
      - "your-team"
    labels:
      - "dependencies"
      - "automated"
```

#### Advanced Configuration with Version Constraints:

```yaml
version: 2
updates:
  - package-ecosystem: "nuget"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
    
    # Ignore specific packages that must remain on fixed versions
    ignore:
      - dependency-name: "Microsoft.AspNetCore.Components.WebAssembly"
        versions: ["7.x", "8.x"]
      - dependency-name: "System.Text.Json"
        update-types: ["version-update:semver-major"]
    
    # Auto-merge minor and patch updates
    allow:
      - dependency-type: "production"
    
    commit-message:
      prefix: "chore"
      include: "scope"
```

### How It Works:
1. **Weekly Checks**: GitHub runs automated dependency scans
2. **Pull Request Creation**: Dependabot creates PRs for outdated packages
3. **Review Process**: Team reviews and approves updates
4. **Automated Merging**: Configure auto-merge for patch/minor updates

### Benefits:
- ✅ Automatic bug fixes and security patches
- ✅ Reduced manual maintenance overhead
- ✅ Stay current with framework updates
- ✅ Version control for all dependency changes

## Additional Automation Opportunities

### 1. Automated Testing Integration
```yaml
- task: DotNetCoreCLI@2
  displayName: 'Run unit tests'
  inputs:
    command: 'test'
    projects: '**/*Tests.csproj'
    arguments: '--configuration $(buildConfiguration) --collect:"XPlat Code Coverage"'
```

### 2. Code Quality Gates
```yaml
- task: SonarCloudPrepare@1
  inputs:
    SonarCloud: 'SonarCloud-Connection'
    organization: 'your-org'
    scannerMode: 'MSBuild'
    projectKey: 'your-project-key'
```

### 3. Notification Integration
```yaml
- task: SlackNotification@1
  condition: always()
  inputs:
    SlackApiToken: '$(SlackToken)'
    MessageAuthor: 'Azure DevOps'
    Channel: '#deployments'
    Message: 'Build $(Build.BuildNumber) - Status: $(Agent.JobStatus)'
```

## Conclusion

Maintaining CI/CD pipelines is an ongoing process that ensures:
- **Reliability**: Automated rollbacks prevent broken deployments
- **Security**: Regular dependency updates patch vulnerabilities
- **Efficiency**: Latest tooling versions provide better performance
- **Robustness**: Staged deployments with proper error handling

By implementing these maintenance practices in Azure DevOps, you create a more resilient and automated deployment pipeline that requires less manual intervention while providing better safeguards for your production applications.
