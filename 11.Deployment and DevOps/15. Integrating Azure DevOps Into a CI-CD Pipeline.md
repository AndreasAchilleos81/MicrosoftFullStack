# Lab: Implementing a CI/CD Pipeline Using Azure DevOps and GitHub

## Overview

This lab demonstrates how to set up a complete CI/CD pipeline that integrates Azure DevOps with GitHub. By the end of this lab, you'll have an automated workflow that triggers builds and deployments whenever code is pushed to your GitHub repository.

## Lab Objectives

- Create an Azure DevOps project
- Establish a GitHub service connection
- Configure a pipeline using YAML
- Trigger automated workflows via Git commits
- Monitor pipeline execution and troubleshoot issues

---

## Step 1: Set Up an Azure DevOps Project

### Navigate to Azure DevOps

1. Go to the Azure DevOps UI at `dev.azure.com`
2. Click **New Project**

### Create the Project

**Project Configuration:**
- **Project Name**: `WeatherCIProject`
- **Visibility**: Choose public or private based on your needs
- Click **Create**

**Example:**
```
Project Name: WeatherCIProject
Description: CI/CD pipeline for weather application
Visibility: Private
Version Control: Git
Work Item Process: Agile
```

### Result

Once created, you'll have access to:
- Repositories
- Pipelines
- Boards
- Test Plans
- Artifacts

## Step 2: Add a GitHub Service Connection

A service connection allows Azure DevOps to access your GitHub repositories securely.

### Navigate to Service Connections

1. Click the **Settings** gear icon (bottom left)
2. Under **Pipelines**, select **Service Connections**
3. Click **New Service Connection**

### Configure GitHub Connection

1. **Select Connection Type**: Choose **GitHub** from the list
2. **Authentication Method**: Select your preferred method
   - OAuth
   - Personal Access Token
   - GitHub App

3. **Authorize Azure DevOps**: 
   - Log in to your GitHub account when prompted
   - Authorize Azure Pipelines to access your repositories

4. **Select Repository**:
   - Choose the specific repository you want to connect
   - For this lab, select your weather application repository

5. **Name the Connection**:
   - Give it a descriptive name (e.g., `GitHub-WeatherApp`)
   - Click **Save**

**Example Configuration:**
```yaml
Connection Name: GitHub-WeatherApp
Authentication: OAuth
Repository: your-username/weather-app
Grant Access: Selected repositories
```

## Step 3: Configure the Azure DevOps Pipeline with YAML

### Navigate to Pipelines

1. Click the **Pipelines** tab in the left sidebar
2. Select **Pipelines**
3. Click **New Pipeline** (or edit existing)

### Create the Pipeline YAML File

Create a file named `azure-pipelines.yaml` (or any name you prefer) in your repository.

### Basic Pipeline Configuration

```yaml
# Trigger: Define when the pipeline runs
trigger:
  branches:
    include:
      - main  # Trigger on any changes to the main branch

# Agent: Specify the build environment
pool:
  vmImage: 'ubuntu-latest'  # Use Ubuntu VM for the build

# Steps: Define the build and deployment tasks
steps:
  # Step 1: Display a welcome message
  - script: echo "Starting CI/CD Pipeline"
    displayName: 'Pipeline Started'

  # Step 2: Install dependencies (example for Node.js)
  - script: |
      npm install
    displayName: 'Install Dependencies'

  # Step 3: Run tests
  - script: |
      npm test
    displayName: 'Run Tests'

  # Step 4: Build the application
  - script: |
      npm run build
    displayName: 'Build Application'
```

### Key Configuration Elements

**1. Trigger Configuration**
```yaml
trigger:
  branches:
    include:
      - main  # Any push to main triggers the workflow
```
- Monitors the `main` branch for changes
- Pipeline runs automatically when code is pushed
- Can add multiple branches or use wildcards

**2. Agent Pool Selection**
```yaml
pool:
  vmImage: 'ubuntu-latest'
```
- Uses Microsoft-hosted Ubuntu agent
- Other options: `windows-latest`, `macOS-latest`
- Can also use self-hosted agents

**3. Steps Definition**
```yaml
steps:
  - script: echo "Hello World"
    displayName: 'Display Message'
```
- Each step performs a specific task
- `displayName` makes logs easier to read
- Can include bash scripts, tasks, or commands

### Complete Example for .NET Application

```yaml
trigger:
  branches:
    include:
      - main

pool:
  vmImage: 'ubuntu-latest'

variables:
  buildConfiguration: 'Release'

steps:
  # Restore NuGet packages
  - task: DotNetCoreCLI@2
    displayName: 'Restore packages'
    inputs:
      command: 'restore'
      projects: '**/*.csproj'

  # Build the project
  - task: DotNetCoreCLI@2
    displayName: 'Build project'
    inputs:
      command: 'build'
      projects: '**/*.csproj'
      arguments: '--configuration $(buildConfiguration)'

  # Run unit tests
  - task: DotNetCoreCLI@2
    displayName: 'Run tests'
    inputs:
      command: 'test'
      projects: '**/*Tests/*.csproj'
      arguments: '--configuration $(buildConfiguration)'

  # Publish the application
  - task: DotNetCoreCLI@2
    displayName: 'Publish application'
    inputs:
      command: 'publish'
      publishWebProjects: true
      arguments: '--configuration $(buildConfiguration) --output $(Build.ArtifactStagingDirectory)'
      zipAfterPublish: true
```

## Step 4: Push Code to GitHub to Trigger the Pipeline

### Prerequisites

Ensure you have:
- A local Git repository initialized
- The remote repository connected to GitHub
- The `azure-pipelines.yaml` file in your repository root

### Git Commands to Trigger the Pipeline

Open your terminal in Visual Studio Code (or any terminal) and execute the following commands:

**1. Add All Files to Staging**
```bash
git add .
```
- Stages all modified and new files
- The `.` represents all files in the current directory

**2. Commit the Changes**
```bash
git commit -m "Add pipeline configuration"
```
- Creates a commit with your changes
- The message describes what was changed

**3. Push to GitHub**
```bash
git push origin main
```
- Pushes your commits to the `main` branch on GitHub
- This action triggers the Azure DevOps pipeline

### Complete Workflow Example

```bash
# Navigate to your project directory
cd /path/to/your/weather-app

# Check the status of your repository
git status

# Stage all changes
git add .

# Commit with a descriptive message
git commit -m "Add pipeline configuration"

# Push to the main branch (triggers pipeline)
git push origin main
```

### What Happens After Push

1. **GitHub receives** your code changes
2. **GitHub webhook** notifies Azure DevOps
3. **Azure DevOps pipeline** detects the trigger condition
4. **Pipeline execution** begins automatically
5. **Build agent** picks up the job
6. **Steps execute** sequentially as defined in YAML

## Step 5: Monitor the Pipeline in Azure DevOps

### Navigate to Pipeline Runs

1. Go to your Azure DevOps project
2. Click **Pipelines** in the left sidebar
3. Select **Pipelines** (not Releases)
4. View the list of recent runs

### Understanding the Pipeline Dashboard

**Pipeline Run List:**
```
Run #23 - ✓ Succeeded - 2m 15s ago - Commit: "Add pipeline configuration"
Run #22 - ✗ Failed - 15m ago - Commit: "Update dependencies"
Run #21 - ✓ Succeeded - 1h ago - Commit: "Fix bug in weather API"
```

### Status Indicators

- **✓ Succeeded**: All steps completed successfully (green)
- **✗ Failed**: One or more steps failed (red)
- **⊙ Running**: Pipeline is currently executing (blue)
- **⊗ Canceled**: Pipeline was manually stopped (gray)

### Investigating Pipeline Runs

**Click on any run** to view detailed information:

**1. Summary View**
- Overall status
- Execution time
- Commit information
- Author details

**2. Jobs and Steps**
```
Job: Build
├─ ✓ Restore packages (15s)
├─ ✓ Build project (45s)
├─ ✓ Run tests (30s)
└─ ✓ Publish application (22s)

Total Duration: 1m 52s
```

**3. Logs**
```
##[section]Starting: Restore packages
Task         : .NET Core
Description  : Build, test, package, or publish a dotnet application
==============================================================
[command]/usr/bin/dotnet restore /home/vsts/work/1/s/WeatherApp.csproj
  Determining projects to restore...
  Restored /home/vsts/work/1/s/WeatherApp.csproj (in 1.2 sec).
##[section]Finishing: Restore packages
```

### Troubleshooting Failed Runs

When a pipeline fails, you can:

1. **Click on the failed run** to see details
2. **Expand the failed step** to view error logs
3. **Review the error message** and stack trace
4. **Make necessary fixes** to your code or YAML
5. **Push changes** to trigger a new run

**Example Failed Step:**
```
##[error]The term 'npm' is not recognized
##[error]PowerShell exited with code '1'
```

**Solution:** Ensure Node.js is installed or add an installation step:
```yaml
- task: NodeTool@0
  inputs:
    versionSpec: '18.x'
  displayName: 'Install Node.js'
```

## Complete Lab Example: Weather Application

### Repository Structure
```
weather-app/
├── azure-pipelines.yaml
├── src/
│   ├── WeatherApp.csproj
│   └── Program.cs
├── tests/
│   └── WeatherApp.Tests.csproj
└── README.md
```

### Full Pipeline YAML
```yaml
# Azure Pipelines configuration for Weather Application
trigger:
  branches:
    include:
      - main
      - develop

pool:
  vmImage: 'ubuntu-latest'

variables:
  buildConfiguration: 'Release'
  projectPath: 'src/WeatherApp.csproj'

stages:
  - stage: Build
    displayName: 'Build and Test'
    jobs:
      - job: BuildJob
        displayName: 'Build Application'
        steps:
          # Checkout code
          - checkout: self
            displayName: 'Checkout repository'

          # Restore dependencies
          - task: DotNetCoreCLI@2
            displayName: 'Restore NuGet packages'
            inputs:
              command: 'restore'
              projects: '$(projectPath)'

          # Build application
          - task: DotNetCoreCLI@2
            displayName: 'Build application'
            inputs:
              command: 'build'
              projects: '$(projectPath)'
              arguments: '--configuration $(buildConfiguration)'

          # Run unit tests
          - task: DotNetCoreCLI@2
            displayName: 'Run unit tests'
            inputs:
              command: 'test'
              projects: 'tests/**/*.csproj'
              arguments: '--configuration $(buildConfiguration) --collect "Code coverage"'

          # Publish application
          - task: DotNetCoreCLI@2
            displayName: 'Publish application'
            inputs:
              command: 'publish'
              projects: '$(projectPath)'
              arguments: '--configuration $(buildConfiguration) --output $(Build.ArtifactStagingDirectory)'
              zipAfterPublish: true
              modifyOutputPath: false

          # Publish build artifacts
          - task: PublishBuildArtifacts@1
            displayName: 'Publish artifacts'
            inputs:
              pathToPublish: '$(Build.ArtifactStagingDirectory)'
              artifactName: 'weather-app-drop'
```

## Best Practices

### 1. YAML Configuration
- Keep YAML files in the repository root
- Use meaningful names for steps and jobs
- Comment complex sections
- Use variables for repeated values

### 2. Branch Strategy
```yaml
trigger:
  branches:
    include:
      - main
      - develop
      - feature/*
    exclude:
      - experimental/*
```

### 3. Error Handling
```yaml
steps:
  - script: |
      npm install || echo "Install failed"
      npm test || echo "Tests failed"
    continueOnError: true
    displayName: 'Run with error handling'
```

### 4. Monitoring
- Review pipeline runs regularly
- Set up notifications for failures
- Monitor execution times
- Archive old runs
