# Monitoring Metrics for Backend System Health and Performance

## Overview

Continuous monitoring is essential for backend websites and APIs to ensure optimal performance and quickly identify issues that need attention. This guide explains how monitoring metrics reflect the health and performance of backend systems, with practical examples using Azure Application Insights.

## Why Monitor Backend Applications?

Once you've created a backend website or API, continuous monitoring enables you to:

- **Ensure optimal performance** - Track response times and resource usage
- **Detect issues early** - Identify errors and anomalies before they impact users
- **Make data-driven decisions** - Use metrics to guide infrastructure and code improvements
- **Understand usage patterns** - Analyze traffic, sessions, and user behavior
- **Plan for scale** - Identify when to upgrade resources based on demand

```csharp
    }

})
.WithName("GetWeatherForecast")

app.Run();

record WeatherForecast(DateOnly Date, int TemperatureC, string? Summary)
```

## Key Metrics and Data Points

### 1. Events Monitoring

Application Insights regularly gathers data about events occurring on your website or API.

**What Events Track:**
- Page views
- API requests
- Custom events
- User interactions
- Dependency calls

**Viewing Events in Azure Portal:**

```bash
# Query events using Azure CLI
az monitor app-insights query \
  --app monitor-demo-insights \
  --analytics-query "
    customEvents
    | where timestamp > ago(1h)
    | summarize count() by name, bin(timestamp, 5m)
    | order by timestamp desc
  " \
  --output table
```

**GitHub Actions - Automated Event Analysis:**

```yaml
name: Analyze Application Events

on:
  schedule:
    - cron: '0 */6 * * *'  # Every 6 hours
  workflow_dispatch:

jobs:
  analyze-events:
    runs-on: ubuntu-latest
    steps:
      - name: Azure Login
        uses: azure/login@v2

        run: |

            --app monitor-demo-insights \
            --analytics-query "
              requests
              | where timestamp > ago(24h)
              | summarize 
                  TotalEvents = count(),
                  UniqueEndpoints = dcount(name),
                  AvgDuration = avg(duration)
              | project TotalEvents, UniqueEndpoints, AvgDuration
            " \
            --output table
      
      - name: Check for Anomalies
        run: |
          # Detect unusual traffic patterns
          az monitor app-insights query \
            --app monitor-demo-insights \
            --analytics-query "
              requests
              | where timestamp > ago(1h)
              | summarize RequestCount = count() by bin(timestamp, 5m)
              | extend Baseline = avg(RequestCount)
              | extend Anomaly = iff(RequestCount > Baseline * 2, 'High', 'Normal')
              | where Anomaly == 'High'
            "
```

**Insights Available:**
- **Traffic sources** - Geographic location and origin
- **Operating systems** - Which OS visitors are using
- **Browser versions** - Browser distribution
- **Endpoint access** - Which endpoints are being hit most frequently
- **Time-based patterns** - Traffic trends over time

### 3. Live Metrics - Real-Time Monitoring

Live Metrics provides a real-time view of your application's performance as events occur.

**Real-Time Metrics Monitored:**
- **Request rate** - Incoming requests per second
- **Request duration** - Average response time
- **Failure rate** - Percentage of failed requests
- **Server resource usage:**
  - Committed memory

  - Exception count

```bash
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --interval PT1M \
  --aggregation Count
```

### 4. Error Monitoring

Tracking errors is critical for maintaining application health and user satisfaction.

**Common Error Types:**
- **500 Internal Server Error** - Application exceptions
- **404 Not Found** - Missing resources
- **401/403 Authorization** - Authentication failures
- **504 Gateway Timeout** - Slow responses

**Example: Simulated Errors in Application**

```csharp
// WeatherForecastController.cs
[HttpGet]
public IActionResult GetForecast()
{
    try
    {
        // Simulate occasional errors
        if (Random.Shared.Next(10) > 7)
        {
            // Simulate memory pressure
            var memoryHog = new byte[1024 * 1024 * 100]; // 100MB
            throw new OutOfMemoryException("Simulated memory issue");
        }
        
        // Simulate variable response times
        Thread.Sleep(Random.Shared.Next(100, 2000));
        
        var forecast = GenerateWeatherForecast();
        return Ok(forecast);
    }
    catch (OutOfMemoryException ex)
    {
        // Application Insights automatically tracks this exception
        return StatusCode(500, "Memory allocation error");
    }
}
```

## Interpreting Metrics for Decision Making

### Scenario 1: High CPU Spike

**Observed Metrics:**
- CPU percentage consistently above 80%
- Request rate has increased 3x
- Response times increasing

**Possible Causes:**
1. **Legitimate traffic surge** - Application popularity or marketing campaign
2. **DDoS attack** - Malicious traffic attempting to overwhelm the server
3. **Inefficient code** - Resource-intensive operations

**Action Steps:**

```yaml
# GitHub Actions - Auto-Scale on High CPU

on:
  workflow_dispatch:
  repository_dispatch:
    types: [high-cpu-alert]

          fi
      
      - name: Notify Team
        uses: actions/github-script@v7
```

## Best Practices for Monitoring Backend Systems

### 1. Establish Baseline Metrics

```bash
# Create baseline report
az monitor app-insights query \
  --app monitor-demo-insights \
  --analytics-query "
    requests
    | where timestamp between(datetime(2025-01-01) .. datetime(2025-01-31))
    | summarize 
        BaselineRequestRate = count() / 30,
        BaselineAvgDuration = avg(duration),
        BaselineP95Duration = percentile(duration, 95),
        BaselineErrorRate = countif(success == false) * 100.0 / count()
  "
```

### 2. Set Up Intelligent Alerting

- Configure alerts based on baseline + standard deviation
- Use dynamic thresholds for metrics that vary by time of day
- Create alert action groups for different severity levels
- Avoid alert fatigue with smart grouping

### 3. Regular Review Schedule

- **Daily**: Check for anomalies and critical errors
- **Weekly**: Review trends and capacity metrics
- **Monthly**: Analyze long-term patterns and optimization opportunities
